# 05 — Build pipeline & diagnostics

Two unrelated subsystems live here. The **build pipeline** (`scripts/build.mjs`) turns the source
tree into a two-file convenience bundle. The **diagnostics system** (`src/core/diagnostics.js`) is
the runtime "actionable console hint" machinery that every layer calls into when it detects a
misconfiguration. They share nothing but this chapter; read the half you need.

---

## Part A — The dist build pipeline

### A.1 What it produces, and why it exists at all

`node scripts/build.mjs` (exposed as `npm run build` in `package.json`) emits exactly two files:

- `dist/simple-admin.js` — `src/index.js` and everything it imports, bundled and minified into one
  ESM file, consumable via a single `<script type="module">`.
- `dist/simple-admin.css` — `src/theme/base.css` + `src/theme/shadcn.css` concatenated and minified
  into one sheet, consumable via a single `<link>`.

The build is **pure convenience**. The library is buildless-by-default: every file under `src/` is
already a valid ES module that runs in the browser as-authored, and `package.json`'s `exports`
points consumers straight at source:

```json
"main": "src/index.js",
"module": "src/index.js",
"exports": {
  ".": "./src/index.js",
  "./theme/*": "./src/theme/*"
}
```

The bundle exists only so a consuming app can reference two URLs instead of pulling the whole source
tree (~65 files). `examples/dist-bundle` is the proof: identical markup to `examples/html-only`, but
imports `dist/simple-admin.js` instead of `src/index.js`, and is expected to behave identically.

The `files` allowlist ships both paths — source *and* bundle — so a published package supports both
the buildless (`import from 'simple-admin'` → `src/index.js`) and bundled workflows:

```json
"files": ["src", "dist", "README.md"]
```

### A.2 The JS build — exact esbuild invocation

```js
const banner = '/*! simple-admin — bundled build, generated by scripts/build.mjs. Do not edit. */\n';

async function buildJs() {
  const result = await build({
    entryPoints: [path.join(srcDir, 'index.js')],
    bundle: true,
    minify: true,
    format: 'esm',
    target: 'es2020',
    outfile: path.join(distDir, 'simple-admin.js'),
    banner: { js: banner },
    legalComments: 'none',
    metafile: true,
  });
  return result.metafile;
}
```

Option by option:

- `bundle: true` — follow the import graph from `src/index.js` and inline it into one file. This is
  the mode where esbuild happily accepts a *single* entry point that transitively pulls in many
  modules; contrast the CSS path below, which cannot use this.
- `format: 'esm'` — output stays an ES module (keeps `export`s intact so the bundle is still a real
  module, not an IIFE/UMD). Matches the "ship plain ES modules" philosophy.
- `target: 'es2020'` — the lowest baseline esbuild is told to keep syntax compatible with. It does
  not down-level below this; the source is already written to modern-evergreen-browser expectations.
- `minify: true` — whitespace/identifier/dead-code minification only. No transformation of language
  semantics.
- `banner: { js: banner }` — prepends the "Do not edit / generated" comment so anyone who opens the
  built file knows it is derived, not a source of truth.
- `legalComments: 'none'` — strips license/legal comment blocks from output. Safe here precisely
  because the library has **zero runtime dependencies** (see A.4): there are no third-party licenses
  that need to survive into the bundle. If a runtime dep were ever added, this choice would need
  revisiting.
- `metafile: true` — esbuild returns a metafile describing the bundle graph. The current script
  captures and returns it but does **not** write it anywhere; it is a hook for future size analysis,
  currently inert.

`sourcemap` is **not set** — there are no source maps produced (see A.4).

### A.3 The CSS build — and why it needs the temp-combine dance

```js
async function buildCss() {
  const files = ['theme/base.css', 'theme/shadcn.css'];
  const chunks = [];
  for (const rel of files) {
    const css = await readFile(path.join(srcDir, rel), 'utf8');
    chunks.push(`/* ---- src/${rel} ---- */\n${css}`);
  }
  const combinedPath = path.join(distDir, '_combined.css');
  await mkdir(distDir, { recursive: true });
  await writeFile(combinedPath, chunks.join('\n\n'), 'utf8');

  await build({
    entryPoints: [combinedPath],
    bundle: false,
    minify: true,
    outfile: path.join(distDir, 'simple-admin.css'),
    banner: { css: banner },
    legalComments: 'none',
  });

  const { unlink } = await import('node:fs/promises');
  await unlink(combinedPath);
}
```

The three-step shape — **read+concatenate → esbuild minify → delete temp** — exists because esbuild
treats CSS entry points differently from JS bundling:

- The JS path uses `bundle: true` with one entry point and lets the `@import`/`import` graph do the
  merging. The CSS here is **not** written as an `@import` chain — `base.css` and `shadcn.css` are
  two independent sibling sheets — so there is no single CSS entry point whose import graph esbuild
  could follow to pull both in.
- esbuild's `entryPoints` with multiple CSS files does not concatenate them into one output the way
  JS bundling merges a dependency graph; each entry point maps to its own output file. There is no
  "multiple CSS inputs → one CSS output" mode analogous to JS `bundle: true` for a set of unrelated
  files. So the script does the concatenation itself in Node (with a `/* ---- src/… ---- */` marker
  comment per file for traceability), writes the result to a single temp file `dist/_combined.css`,
  and hands esbuild that **one** entry point with `bundle: false` (minify-only, no graph walking).
- The temp file is unlinked afterward so it never becomes a stray artifact in `dist/`. The delete is
  the last step of `buildCss` and is not wrapped in a `finally` — if esbuild throws, `_combined.css`
  can be left behind (a minor gotcha; see A.6).

Note the banner is applied via `banner: { css: banner }` (CSS comment channel) — the same "Do not
edit" string, correctly emitted as a CSS comment rather than JS.

### A.4 What the build deliberately does NOT do

There is no TypeScript stripping, no JSX transform, no Babel, no polyfilling, and **no source maps**
(no `sourcemap` option appears in either `build()` call). This is intentional and consistent with the
buildless-by-default design:

- The source is plain, standards-compliant ES2020 JavaScript. It needs no transformation to run —
  esbuild is invoked in `minify`/`bundle` mode, never in a transpile-a-different-language mode.
- Because the shipped source *is* the runnable source (`exports` → `src/index.js`), a source map on
  the bundle would map minified output back to a concatenation the developer never edits anyway; the
  developer-facing debugging story is "import `src/` directly, no build," which needs no map.
- **esbuild is a `devDependency` only** and is **never imported by library source**. Verified by the
  `devDependencies` block and by the header comment in `build.mjs`:

  ```json
  "devDependencies": {
    "esbuild": "^0.28.1",
    "playwright": "^1.61.1"
  }
  ```

  The zero-runtime-dependencies promise therefore holds regardless of build tooling: nothing under
  `src/` reaches for esbuild, playwright, or any package. You could delete `node_modules` entirely
  and the library still runs; only the (optional) build and (optional) browser tests would stop
  working.

### A.5 Operational notes — real gotchas encountered in this environment

From `_docs/verification-plan.md`, two environment-specific blockers were hit and worked around;
document them so the next maintainer does not burn time re-discovering them:

1. **`npm` segfaults (exit 139) on every invocation, including `npx`, in the build sandbox.** The
   workaround was to run the build through **`bun`**: `bun add -D esbuild`, `bun add -D playwright`,
   and `bun run build` all work. `pnpm` is an equally valid alternative. The build script itself is
   plain Node ESM — `node scripts/build.mjs` — so any runner that can install `esbuild` and invoke
   Node works; the segfault is an `npm`-in-this-sandbox problem, not a code problem.
2. **Chromium cannot launch under the sandbox's macOS Seatbelt profile** (`bootstrap_check_in …
   Permission denied (1100)` — a mach-port IPC rendezvous denied at the OS level, independent of
   Playwright's `--no-sandbox`). This only affects the *browser verification* step
   (`scripts/verify-browser.mjs`), not the build. On a normal dev machine / CI it is a non-issue;
   Playwright `^1.61.1` and a cached Chromium are already installed as devDependencies.

For the record, the last known-good build output was `dist/simple-admin.js` ≈ 99.4 KB and
`dist/simple-admin.css` ≈ 8.7 KB (minified).

### A.6 Build-script surprises worth knowing

- **`dist/` does not need to pre-exist.** `main()` calls `await mkdir(distDir, { recursive: true })`
  up front, and `buildCss()` does the same defensively. A clean checkout builds fine.
- **The build does NOT clean stale files first.** There is no `rm -rf dist` step. `simple-admin.js`
  and `simple-admin.css` are overwritten by `outfile`, but any *other* file you dropped in `dist/`
  (an old artifact, a manually-added file) survives. If you rename an output, the old one lingers.
- **The temp `dist/_combined.css` is only cleaned on the success path.** If esbuild throws during the
  CSS minify, the `unlink` never runs and the temp file is left in `dist/`. `main()` catches the
  error, logs it, and `process.exit(1)`s — but the stray temp is not swept.
- **`metafile` is captured but unused** — `buildJs` returns it and `main` discards it. Not a bug,
  just a dangling seam.
- The final `report()` calls just `stat` the two outputs and print KB sizes; they are cosmetic.

## Part B — The diagnostics / console-hint system

`src/core/diagnostics.js` is a small, self-contained module (~150 lines, zero imports) implementing
the "warn, don't throw" contract: when any layer detects a misconfiguration it calls
`diagnostics.warn(code, detail)` or `diagnostics.error(code, detail)`, which formats a
prefixed-and-actionable message, dedupes it, and routes it to `console.warn`/`console.error` subject
to a global log level.

### B.1 Public surface

```js
export const setLogLevel = (level) => { … };   // 'silent' | 'error' | 'warn' | 'verbose'
export const getLogLevel = () => …;             // current level as a string
export const warn = (code, detail) => emit(LEVELS.warn,  console.warn,  code, detail);
export const error = (code, detail) => emit(LEVELS.error, console.error, code, detail);
export const _resetDiagnostics = () => seen.clear();   // test/tooling seam
```

Every message is prefixed `[simple-admin]` and is written to (1) name the element/resource/source at
fault, (2) state the likely cause, (3) state the fix.

### B.2 Log levels and gating — does `error` ever get suppressed?

```js
const LEVELS = { silent: 0, error: 1, warn: 2, verbose: 3 };
let currentLevel = LEVELS.warn;   // default
```

`setLogLevel(level)` ignores unknown strings (`if (!(level in LEVELS)) return;`) — a typo is a no-op,
not a reset. `getLogLevel()` reverse-maps the numeric level back to its string name.

Gating happens in `emit`:

```js
const emit = (minLevel, consoleMethod, code, detail) => {
  if (currentLevel < minLevel) return;
  const message = format(code, detail);
  if (seen.has(message)) return;
  seen.add(message);
  consoleMethod(message);
};
```

`warn` passes `minLevel = LEVELS.warn (2)`, `error` passes `minLevel = LEVELS.error (1)`. So:

- Default level `warn (2)`: both warnings and errors print.
- `setLogLevel('error')` → level `1`: **errors still print, warnings are suppressed** (`2 > 1`).
- `setLogLevel('silent')` → level `0`: **everything, including errors, is suppressed** (`1 > 0` and
  `2 > 0`). `silent` is the only level that muzzles `error`.
- `setLogLevel('verbose')` → level `3`: same output as `warn` today — no code emits at a "verbose"
  min-level, so `verbose` is currently a synonym for `warn` in effect, reserved for future use.

So `error` is *not* categorically un-suppressable — it is suppressed only at `silent`. Every level
except `silent` shows errors; `warn` and `verbose` additionally show warnings.

### B.3 Message templates — storage, lookup, formatting

Templates live in one object literal, `TEMPLATES`, keyed by diagnostic code, each a function of the
`detail` object:

```js
const TEMPLATES = {
  'field-missing-source': (d) =>
    `<${d.tag}> is missing the required "source" attribute (inside ${d.ctx}). ` +
    `Add source="fieldName" so it knows which record property to display. Skipping.`,
  // …one entry per code…
};
```

`format` resolves the final string:

```js
const format = (code, detail = {}) => {
  if (detail && typeof detail.message === 'string') return detail.message;
  const template = TEMPLATES[code];
  const body = template ? template(detail) : `${code} ${safeDetail(detail)}`;
  return `[simple-admin] ${body}`;
};
```

Three important behaviors:

1. **`detail.message` escape hatch.** If a caller passes an explicit `detail.message` string, it is
   returned **verbatim** — the `[simple-admin]` prefix is *not* re-added, so custom callers must
   include it themselves. This is how `components/admin.js` emits its JS-config "unknown field/input
   type" hints: it reuses the `unknown-element` code but supplies a fully-formed `message` (already
   prefixed) rather than relying on the template.
2. **Unknown codes degrade, don't crash.** A code with no template falls back to
   `` `${code} ${safeDetail(detail)}` `` where `safeDetail` is a try/catch'd `JSON.stringify`.
3. Helper `previewRecord(record)` (used by `record-missing-id`) JSON-stringifies and truncates a
   record to ≤60 chars so a huge object isn't dumped into the console.

### B.4 The dedupe algorithm — exact key, structure, lifetime

```js
const seen = new Set();
// inside emit():
if (seen.has(message)) return;
seen.add(message);
```

- **Key: the fully-formatted final message string** — the output of `format(code, detail)`, i.e.
  after template interpolation and prefixing. Not the code, not `code + detail`. Two calls with the
  same `code` but different `detail` that render to different strings both print; two calls that
  render to the *same* string print once. This is deliberate: a 100-row datagrid with one
  misconfigured column produces one identical message 100 times → logged once. But the *same* code
  fired for two different resources/tags produces two different strings → both logged (you want to
  hear about each distinct misconfiguration).
- **Structure: a module-level `Set<string>`**, `seen`. It is process-global (one per module
  instance) and lives for the lifetime of the page.
- **Lifetime / clearing: it never clears on its own.** The only reset is the exported
  `_resetDiagnostics()` (`seen.clear()`), a test/tooling seam — call it between tests so a message
  asserted in one test doesn't get swallowed as a dup in the next. Changing the log level does
  **not** clear `seen`; a message suppressed under `silent` and then re-fired after
  `setLogLevel('warn')` will still print (it was never added to `seen` while suppressed, because the
  level check short-circuits *before* the `seen` check).

#### How this differs from the design doc's speculative §11 list

`_docs/react-admin/10-simple-admin-architecture.md` §11 was written before implementation and listed
~13 situations. The finished implementation has **20 distinct codes**. Everything in the design doc's
table shipped, but **seven codes were added during implementation** for preconditions the design doc
didn't anticipate:

- `field-no-record-context` — field with a source but no record context ancestor.
- `input-no-form` — input outside any form/filter host.
- `field-reregistered` / `input-reregistered` — double registration of a custom field/input type.
- `list-no-body` — `<sa-list>` with no datagrid/simple-list.
- `datagrid-no-list` — `<sa-datagrid>` outside a `<sa-list>`.
- `pagination-no-list` — `<sa-pagination>` outside a `<sa-list>`.

The design doc's `route-view-missing` also grew a *second* call site (error variant for an entirely
unknown route resource, in addition to the warn variant for a known resource missing a view). Treat
this chapter's table as the source of truth, not §11.

### B.6 Design rationale — "warn, don't throw"

The pervasive philosophy is **degrade gracefully, log actionably**: a single misconfigured field
must never take down the whole admin page. Every precondition check chooses to warn (or error) and
render a safe fallback rather than throw. Three concrete examples straight from source:

1. **`fields/baseField.js`** — a field missing its `source`:
   ```js
   if (!this.source && this._descriptor.type !== 'function') {
     diagnostics.error('field-missing-source', { tag: this.localName, ctx: this._contextLabel() });
     return; // degrade: render nothing, never throw.
   }
   ```
2. **`inputs/baseInput.js`** — an input that can't find a FormStore:
   ```js
   this._form = host ? host.formStore : null;
   if (!this._form) {
     diagnostics.warn('input-no-form', { tag: this.localName, source: this.source });
     return;
   }
   ```
   The input simply doesn't register/bind — the rest of the form still works.
3. **`fields/referenceField.js`** — a reference to an undeclared resource renders the raw id instead
   of failing to resolve:
   ```js
   if (!hasResource(this.reference)) {
     diagnostics.warn('reference-undeclared', { reference: this.reference, resource: this._resourceName() });
     this._renderFallback(value);
     return;
   }
   ```

The pattern is always: detect precondition → emit one actionable hint (naming the tag/source/
resource and the exact fix) → render a safe fallback and `return`. The browser-verification plan even
codifies this: `[simple-admin]` console warnings/errors are **expected and fine**; the check fails
only on an uncaught `pageerror`.

### B.7 Gotchas for adding a new diagnostic code

To add a code correctly:

1. **Add a template** to the `TEMPLATES` object in `src/core/diagnostics.js`, keyed by your new
   code, as a `(detail) => string` function. Follow the four-part contract: prefix is added
   automatically by `format`, so the template body should *not* include `[simple-admin]` — name the
   element/resource/source at fault, the likely cause, and the fix. (Only pass a pre-formatted,
   pre-prefixed `detail.message` if you're bypassing the template on purpose, as `admin.js` does for
   its JS-config hints.)
2. **Call `diagnostics.warn(code, detail)` or `.error(code, detail)`** at the site, passing exactly
   the `detail` keys your template reads. Choose `warn` vs `error` by the level-gating semantics in
   B.2: use `error` for "nothing can work" conditions you want visible even at `setLogLevel('error')`
   (data/auth provider missing, missing `source`, missing provider method); use `warn` for
   "degrades but page still works" conditions.
3. **You do NOT need to touch `setLogLevel`.** Levels are fixed (`silent/error/warn/verbose`); a new
   code just piggybacks on `warn`/`error`. There is no per-code configuration.
4. **Dedup implication:** dedup keys on the *rendered string*. If your template interpolates enough
   `detail` (tag, source, resource) to make each distinct misconfiguration render distinctly, each
   prints once; if your template is constant (like `datagrid-no-list`, which ignores `detail`), it
   prints exactly once per page no matter how many elements trip it. Design the template's
   granularity to match how many *distinct* instances you want the user to hear about.
5. **Tests:** call `_resetDiagnostics()` between tests to clear the `seen` set, since it never clears
   on its own and a message logged in one test will be silently deduped in the next otherwise.
